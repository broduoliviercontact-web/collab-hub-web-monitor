# Release v1.1.1 — Collab-Hub Web Monitor — Live Audio Edition (correctif multi-listener)

Date : 2026-07-11. **Patch** de v1.1.0. Tag `v1.1.1`. Release GitHub « latest
stable ».

## Nature du patch

Correctif d'un bug runtime v1.1.0 : un **second listener** rejoignant une room
où le performer diffuse déjà restait bloqué en `waiting_for_track` (aucun son,
aucune erreur, `trackSid=null`, `performer=null`). Le premier listener (qui
était déjà connecté quand le performer a démarré) n'était pas affecté.

## Cause racine

Les **publications audio déjà publiées au moment de `connect()`** n'étaient pas
rattachées. `TrackSubscribed` peut se déclencher pendant la résolution de
`room.connect()`, avant que les handlers applicatifs ne soient effectivement en
place (race) → la piste pré-existante est perdue → `waiting_for_track` permanent.

`roomID` n'est pas en cause : les deux rôles reçoivent `room: 'main'`
(`api/livekit/token.js`). `participants=0` est attendu : `ParticipantConnected`
ne se déclenche que pour les arrivées *postérieures*, pas pour les participants
déjà présents.

## Correctif (`src/livekit/livekitListener.js`)

- Après `await room.connect()`, nouvelle fonction `attachExistingAudioTracks()`
  qui inspecte `room.remoteParticipants` → `trackPublications` et rattache
  toute publication audio dont `.track` est déjà présent (pattern documenté
  LiveKit pour les pistes pré-existantes).
- **Idempotent** avec `onTrackSubscribed` via `selectProgramTrack` (la piste
  déjà courante est ignorée → pas de double attach).
- `setState('waiting_for_track')` post-connect ne se déclenche que si aucune
  piste n'a été rattachée (`if (!currentTrack)`).
- Vérifié contre l'API réelle `livekit-client` : `Room.remoteParticipants` est
  un `Map<string, RemoteParticipant>`, `RemoteParticipant.trackPublications`
  un `Map<string, RemoteTrackPublication>`, `TrackPublication.kind` =
  `Track.Kind` (`"audio"`), `.track` défini quand souscrit.

## Vérifications

- **330 tests `node:test`** verts (326 → 330, +4) :
  - 2e listener, performer déjà publié, **aucun** `TrackSubscribed` → piste
    rattachée (`hasAudioTrack`/`audioTrackSid`/`performerIdentity` + `playing`) ;
  - `TrackSubscribed` tardif + piste existante → pas de double rattachement
    (`attachTrack` × 1) ;
  - participant déjà présent → `participants=0` mais piste rattachée ;
  - aucun participant → `waiting_for_track` (comportement inchangé).
- `npm run check` vert (tests + licence + tracked + secrets + maxpat + build).
- Build Vite propre (chunk-split inchangé : gate 16 ko avant auth,
  controlRoomPage 529 ko post-auth, main 51 ko).
- CI verte sur le commit de release.
- **Validation runtime** : multi-listener validé (Chrome performer, Chrome
  listener, Safari listener, BlackHole, LiveKit, session Control Room, bouton
  enceinte + −20 dB) — cf. v1.1.0.

## Compatibilité

Compatibilité **totale** avec v1.1.0 côté métadonnées et audio. Aucune API
publique changée. Même patch Max, même protocole Collab-Hub, mêmes cinq champs.

## Sécurité

Aucun changement de sécurité vs v1.1.0 (session Control Room signée HMAC-SHA256,
secrets serveur uniquement, timing-safe, etc.).

## Limites connues (inchangées vs v1.1.0)

- chunk `controlRoomPage` > 500 ko (avertissement Vite, `manualChunks` non
  configuré) ;
- pas de refresh token (TTL 2 h ; reconnexion native suffit en session courte) ;
- changement de device en capture non supporté (sélecteur désactivé en
  capture) ;
- double performer non protégé (les deux peuvent publier `program-audio` ; le
  listener conserve la première piste, ignore la seconde) ;
- wiring de `check-livekit-secrets` dans la CI GitHub reporté ;
- vulnérabilités `npm audit` (esbuild/vite, devDeps, préexistantes) non corrigées.
