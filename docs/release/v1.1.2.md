# Release v1.1.2 — Collab-Hub Web Monitor — Live Audio Edition (correctif listener multi-client + iOS)

Date : 2026-07-11. **Patch** de v1.1.1. Tag `v1.1.2`. Release GitHub « latest
stable ».

## Nature du patch

Correctif d'un bug runtime persistant après v1.1.1 : un listener se connecte
correctement à `room main` (même `roomID`), mais termine en `waiting_for_track`,
`participantCount=0`, `trackSid=null`, `performer=null`, **aucune erreur**.
Un autre listener pouvait parfois jouer correctement (variabilité de timing) ;
iOS ne jouait pas. Le correctif v1.1.1 (rattachement des pistes déjà publiées)
ne couvrait pas la race `TrackSubscribed` pendant `connect()` ni la
souscription explicite des publications non souscrites.

## Cause racine

Quatre défauts convergents dans `src/livekit/livekitListener.js` :

1. **Garde `!isConnected` sur `onTrackSubscribed`** : `TrackSubscribed` peut
   se déclencher pendant `await room.connect()`, avant que l'état passe à
   `connected`. Le handler rejetait l'évènement → la piste pré-existante ou
   arrivant tôt était perdue.
2. **Pas de réconciliation post-connect des participants existants** : un
   participant déjà publié avant l'arrivée du listener n'était jamais scanné.
3. **Pas de souscription explicite** : une `RemoteTrackPublication` avec
   `track === null` (publiée, non souscrite) restait en l'état — jamais
   `setSubscribed(true)`.
4. **`participantCount` incrémenté seulement sur `ParticipantConnected`** : un
   participant déjà présent n'était pas compté → `participantCount=0` malgré un
   performer connecté.

**Évènement perdu** : `RoomEvent.TrackSubscribed` tiré par le SDK pendant la
fenêtre `connecting` de `room.connect()` — rejeté par le garde `!isConnected`.
La piste n'était jamais attachée → `waiting_for_track` permanent, sans erreur.
La variabilité inter-listeners vient du moment exact de l'évènement vs la
résolution de `connect()` (microtask ordering).

## Correctif (`src/livekit/livekitListener.js`)

- **Garde `isConnected` supprimé** : `onTrackSubscribed` devient
  `if (destroyed || !room || userStopped) return;`. La piste est acceptée dès
  que la Room existe, y compris pendant `connecting`. Reste idempotent via
  `selectProgramTrack`.
- **`reconcileRemoteParticipants()`** (pure, testable via
  `scanRemoteAudio(room, trackKinds)`) : après `room.connect()`,
  `ParticipantConnected`, `TrackPublished`, `TrackSubscriptionStatusChanged`,
  `Reconnected`, plus retry borné `[100, 300, 750] ms` puis arrêt (pas de
  polling infini). Recalcule `participantCount` depuis
  `room.remoteParticipants.size`, itère tous les participants et leurs
  `audioTrackPublications`, priorise program-audio (deux passes), attache
  `pub.track` si présent. Idempotent (pas de double attach) ; anti-spam via
  un `Set` de `requestedSubscriptions`.
- **`setSubscribed(true)` explicite** : pub audio avec `track === null` →
  `pub.setSubscribed(true)` si la méthode existe, puis attente de
  `TrackSubscribed`. Nouveaux évènements câblés : `TrackPublished`,
  `TrackSubscriptionStatusChanged`, `AudioPlaybackStatusChanged`.
- **`participantCount` corrigé** : `room.remoteParticipants?.size ?? 0` après
  connexion et à chaque réconciliation (pas seulement `ParticipantConnected`).
- **iOS/Safari `room.startAudio()`** : `startAudio()` appelle
  `room.startAudio()` (instance method de `livekit-client ^2.20.1`) dans le
  geste utilisateur puis `audioSink.play()` ; si pas de piste encore,
  `audioUnlocked=true`, attachement + replay au `TrackSubscribed` suivant.
  **Fallback ACTIVER LE SON** : si Safari refuse l'autoplay
  (`NotAllowedError`) → `waiting_for_user` + `autoplayBlocked=true` + bouton
  dédié ; second geste → `room.startAudio()` + `play()` → `playing`.
- **Microtask ordering** : garde `if (state === 'connecting') setState('connected');`
  — on ne régresse pas un état déjà avancé (évite l'écrasement de `playing` par
  la continuation de `connect()`).
- **Diagnostics** (snapshot + panneau `?debug=1`, **aucun token/secret**) :
  `audioUnlocked`, `roomCanPlaybackAudio`, `existingParticipants`,
  `existingAudioPublications`, `subscribedAudioPublications`,
  `reconciliationCount`, `lastTrackEvent`, `lastTrackPublishedAt`,
  `lastTrackSubscribedAt`. Vérifié contre l'API réelle `livekit-client`.

## Vérifications

- **350 tests `node:test`** verts (330 → 350, +20) : `scanRemoteAudio` (4) +
  F.1–F.15 couvrant les 12 cas obligatoires (race `TrackSubscribed` pendant
  `connect()`, participant présent avant connect, pub audio avec piste
  immédiate, pub `track=null` → `setSubscribed(true)`, `TrackSubscribed` après
  réconciliation, 2 listeners successifs, `Reconnected` relance la
  réconciliation, pas de double attach, iOS `room.startAudio()` dans le geste,
  autoplay refusé → `waiting_for_user` + bouton ACTIVER LE SON, second geste →
  `playing`, `participantCount` reflète les `remoteParticipants` existants) +
  retry borné + anti-spam + snapshot sans secret.
- `npm run check` vert (tests + licence + tracked + secrets + maxpat + build).
- Build Vite propre (chunk-split inchangé : gate ~16 ko avant auth,
  controlRoomPage ~529 ko post-auth, main ~51 ko).
- CI verte sur le commit de release.
- **Validation runtime** : 3 listeners simultanés sur réseaux différents
  (ordinateur Wi-Fi, partage de connexion, iOS) — tous se connectent et
  entendent le stream ; multi-listener OK, iOS OK, performer déjà ON AIR avant
  connexion OK, Track SID reçu, participant performer détecté, pas de double
  audio, aucune erreur bloquante.

## Compatibilité

Compatibilité **totale** avec v1.1.1 / v1.1.0 / v1.0.1. Aucune API publique
changée. Même patch Max, même protocole Collab-Hub, mêmes cinq champs.

## Sécurité

Aucun changement de sécurité vs v1.1.1 (session Control Room signée
HMAC-SHA256, secrets serveur uniquement, timing-safe, contrôle
`check-livekit-secrets`). Aucun nouveau secret ; les diagnostics ne contiennent
ni token, ni mot de passe, ni valeur d'environnement.

## Limites connues (inchangées vs v1.1.0)

- chunk `controlRoomPage` > 500 ko (avertissement Vite, `manualChunks` non
  configuré) ;
- pas de refresh token (TTL 2 h ; reconnexion native suffit en session courte) ;
- changement de device en capture non supporté (sélecteur désactivé en
  capture) ;
- double performer non protégé (les deux peuvent publier `program-audio` ; le
  listener conserve la première piste, ignore la seconde) ;
- wiring de `check-livekit-secrets` dans la CI GitHub reporté ;
- vulnérabilités `npm audit` (esbuild/vite, devDeps, préexistantes) non corrigées.
