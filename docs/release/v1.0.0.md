# Release v1.0.0 — Collab-Hub Web Monitor

Date : 2026-07-10.

## Présentation du projet

Page web publique affichant en temps réel le contexte du morceau diffusé —
**titre**, **auteur**, **sous-titre**, **note descriptive** et **lien**
optionnel — piloté depuis Max/MSP via Collab-Hub (Socket.IO). Une fiche
programme / cartouche éditoriale sobre, sans rechargement.

## Architecture

Client Vite + JavaScript vanilla (pas de React). `socket.io-client` est la
seule dépendance runtime. SPA statique, sans backend, sans fonctions
serverless : Vercel ne sert que les assets ; le navigateur se connecte
directement à Collab-Hub par WebSocket.

```
src/
├── main.js                      # point d'entrée ; socket -> état -> rendu
├── collabHub/
│   ├── socketClient.js          # connexion Socket.IO, fallback auth, observation
│   ├── observeGuard.js          # observation idempotente par socket.id
│   ├── authMode.js              # résolution du mode d'auth (anonymous/guest)
│   └── messageRouter.js         # normalisation + routage (KNOWN/HEARTBEAT headers)
├── state/
│   ├── soundState.js            # état courant des 5 champs
│   ├── persist.js               # persistance locale localStorage
│   └── freshness.js             # fraîcheur + heartbeat
├── ui/
│   ├── renderSoundInfo.js       # rendu DOM + sécurité du lien
│   └── renderConnectionStatus.js# indicateur de connexion
├── diagnostic/
│   └── diagnosticPanel.js       # panneau ?debug=1 (chunk dynamique)
└── styles/
    └── main.css                 # design sombre, responsive, reduced-motion
max/
├── CollabHub_Web_Text_Sender.maxpat  # patch Max émetteur
├── validate-maxpat.mjs               # validateur statique du patch
└── README.md                         # doc Max détaillée
test/runTests.mjs                 # 48 tests (node:test, zéro dépendance)
.github/workflows/ci.yml          # CI GitHub Actions
```

Séparation connexion · routage · état · rendu · styles. Les modules purs
(router, state, render, persist, freshness) ne dépendent ni d'`import.meta.env`
ni du `document` global — testables en Node.

## Prérequis

- Node.js 24 (recommandé ; CI l'utilise) pour le build web.
- Max 8 ou ultérieur (patch enregistré en format Max 9, rétro-compatible).
- Package **Collab-Hub Max Client** installé (voir `max/README.md`).
- Accès Internet vers `https://server.collab-hub.io`.

## Installation web

```bash
git clone <dépôt>
cd collab-hub-web-monitor
npm install
cp .env.example .env       # vérifier VITE_COLLAB_HUB_NAMESPACE=hub
npm run dev                # http://localhost:5173
```

Build de production :

```bash
npm run build     # sortie dans dist/
npm run preview   # prévisualiser le build
```

## Installation Max

1. Installer le **client Collab-Hub Max Client** (dépôt
   `Collab-Hub-io/Collab-Hub-Max-Client`). Le patch instancie l'abstraction
   officielle `ch.client.maxpat`. Deux options détaillées dans `max/README.md` :
   - Option A : ajouter le dossier racine du dépôt au **Search Path / File
     Preferences** (cocher Subfolders) ;
   - Option B : placer le package dans `~/Documents/Max 8/Packages/`.
2. Ouvrir `max/CollabHub_Web_Text_Sender.maxpat`.

Vérification : créer un objet `ch.client` — l'auto-complétion doit le proposer.

## Configuration `.env.local`

Copier `.env.example` en `.env` (développement local). Aucun secret.

| Variable | Rôle | Défaut |
|---|---|---|
| `VITE_COLLAB_HUB_URL` | Serveur Collab-Hub (sans namespace ni slash final) | `https://server.collab-hub.io` |
| `VITE_COLLAB_HUB_NAMESPACE` | Namespace Socket.IO — **doit valoir `hub`** | `hub` |
| `VITE_COLLAB_HUB_AUTH_MODE` | `anonymous` (socket direct) ou `guest` | `anonymous` |
| `VITE_DIAG_ONANY` | (diagnostic) `onAny` au démarrage | `1` |

## Variables Vercel

Configurées dans le projet Vercel (Production + Preview). Cuites dans le
bundle au build, aucune n'est secrète.

| Variable | Valeur |
|---|---|
| `VITE_COLLAB_HUB_URL` | `https://server.collab-hub.io` |
| `VITE_COLLAB_HUB_NAMESPACE` | `hub` |
| `VITE_COLLAB_HUB_AUTH_MODE` | `anonymous` |

`VITE_COLLAB_HUB_NAMESPACE` est **obligatoire** : sans elle le bundle
construit avec un namespace vide (racine `/`) et ne reçoit aucun contrôle
publié par Max (qui utilise `/hub`).

## Utilisation des cinq champs

1. Lancer le site web (local `npm run dev` ou production Vercel) → « Connecté » ;
   le `serverMessage` annonce `Collab-Hub Version: 0.3.4. You're in Namespace
   /hub`.
2. Ouvrir le patch Max, connecter le CH-Client à
   `https://server.collab-hub.io`, attendre la version 0.3.4 (identique côté
   Max et côté web).
3. Remplir les cinq champs dans Max (valeurs **entre guillemets** dans les
   boîtes message).
4. Cliquer **ENVOYER LES 5 CHAMPS** : le bouton envoie chaque champ deux fois
   (register puis deliver 300 ms plus tard) → les blocs se remplissent en
   temps réel.
5. Modifier un seul champ dans Max → seul ce champ change sur la page
   (isolation).
6. URL valide (`https://…`) → lien « En savoir plus » ; `javascript:` ou vide →
   lien masqué.

## Fonctionnement du heartbeat

Le patch publie `publish all sound_heartbeat 1` toutes les 10 s tant que le
CH-Client est connecté (zone HEARTBEAT, `metro 10000` piloté par `connected`).

- **`Connecté — Max actif`** : serveur connecté ET heartbeat reçu depuis < 25 s
  (`MAX_ACTIVE_THRESHOLD_MS`).
- **`Connecté — Max silencieux`** : serveur connecté mais heartbeat > 25 s
  (ou jamais reçu).
- **`Reconnexion…` / `Déconnecté`** : le statut serveur est prioritaire.

Le heartbeat est un **header technique** : jamais affiché comme contenu, jamais
persisté, ne déclenche pas l'animation des blocs. Au rechargement,
`maxLastSeenAt` n'est pas restauré → Max reste « silencieux » jusqu'au premier
heartbeat réellement reçu (par conception, anti-tromperie). La fraîcheur du
contenu (`CONTENT_FRESH_THRESHOLD_MS = 300000`, 5 min) pose l'attribut
`data-content-fresh` sur la carte ; un contenu ancien est légèrement estompé,
jamais masqué.

Logique dans `src/state/freshness.js` (pur, horloge injectable).

## Limites connues

- La **1re publication** d'un header l'enregistre seulement et ne pousse pas la
  valeur ; le patch envoie chaque champ deux fois (register + deliver).
- Persistance **locale par navigateur** uniquement : pas d'historique, pas de
  multi-postes. Données corrompues/incompatibles → valeurs par défaut.
- `maxLastSeenAt` non restauré après rechargement → Max « silencieux » jusqu'au
  prochain heartbeat (≤ 10 s).
- Le serveur public v0.3.4 est vieillissant ; un changement d'URL/protocole
  imposerait d'ajuster `.env`/Vercel.
- L'auth invitée `/api/v1/auth/guest` est absente du serveur v0.3.4 (404) ; le
  mode `anonymous` (défaut) n'émet aucune requête d'auth. `guest` n'est
  pertinent que sur un serveur v0.5.
- `observedControls` côté serveur peut contenir des entrées `null` / doublons
  de usernames (comportement du serveur public 0.3.4) ; on ne peut pas les
  nettoyer depuis le client — on évite seulement d'en créer via l'observation
  idempotente.
- Le diagnostic `?debug=1` est accessible à quiconque connaît l'URL (sans
  login) — acceptable pour un spike, à protéger en production ultérieure.
- **Aucune licence** n'est définie dans le dépôt ; le ZIP de release Max
  n'inclut donc pas de fichier `LICENSE` (voir statut licence ci-dessous).
