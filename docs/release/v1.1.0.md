# Release v1.1.0 — Collab-Hub Web Monitor — Live Audio Edition

Date : 2026-07-10. **Édition audio en direct** (LiveKit). Tag `v1.1.0`.

> **Statut de validation au moment de la préparation** : cette release a été
> préparée (bump 1.1.0, docs, package, commit, CI) avec une validation
> **automatisée + HTTP production** complète. Les **tests runtime physiques**
> (Ableton+BlackHole, écoute réelle, Chrome/Safari/mobile, coupure réseau,
> devicechange, qualité/latence audio, double performer) sont **à valider
> manuellement** avant publication « latest stable » — voir
> `docs/bmad/13-livekit-stabilization-and-release.md` et la checklist GO/NO-GO
> ci-dessous. Aucun résultat runtime n'est présenté comme validé dans ce document.

## Nature de la release

La v1.1.0 ajoute la couche audio temps réel LiveKit (Control Room performer +
écoute publique), par-dessus la fiche programme Collab-Hub inchangée depuis
v1.0.1. **Aucun changement du protocole Collab-Hub, aucun changement du patch
Max, aucun changement des cinq champs.** Le flux métadonnées Max → site reste
identique.

### Vérifié par HTTP production (déploiement Vercel)

- `GET /` → 200 ; `GET /control-room` → 200 (titre « Control Room ») ;
- `GET /api/livekit/token` → 405 (`Allow: POST`) ;
- `POST /api/livekit/token {role:"listener"}` → 200, réponse contient `token` +
  `room:"main"` (valeur du token non affichée ici) ;
- `Cache-Control: no-store` sur la réponse 200 ;
- `POST /api/livekit/token {role:"performer", password:<mauvais>}` → 401 ;
- `POST /api/livekit/token {role:<invalide>}` → 400 ;
- les 4 variables serveur (`LIVEKIT_URL`, `LIVEKIT_API_KEY`,
  `LIVEKIT_API_SECRET`, `PERFORMER_PASSWORD`) sont configurées en production
  (sinon l'endpoint renverrait 503) et `PERFORMER_PASSWORD` ≥ 8 caractères ;
- `VITE_LIVEKIT_ENABLED=true` confirmé en production (bundle public référence
  `listenerSection` / `DIRECT AUDIO` / `lk-listener` → non éliminé au build) ;
- route `/control-room` + chunk `controlRoomPage` live en production.

### Vérifié par revue de code (propriétés, sans exécution runtime)

- grants LiveKit corrects : performer `canPublish:true`,
  `canSubscribe:false`, `canPublishData:false` ; listener `canPublish:false`,
  `canSubscribe:true`, `canPublishData:false` ;
- TTL token 2 h (7200 s) ;
- contraintes capture « musique » : `echoCancellation:false`,
  `noiseSuppression:false`, `autoGainControl:false`, `channelCount:{ideal:2}`
  avec échelle de fallback ;
- publisher : `dtx:false`, `forceStereo:true`,
  `audioPreset:AudioPresets.musicHighQualityStereo` (128 kbps) ;
- graphe Web Audio : aucune connexion à `context.destination` → pas de
  monitoring local (anti-Larsen) ;
- `AudioContext.resume()` appelé au geste utilisateur (compat Safari) ;
- labels devices débloqués via stream jetable après permission (compat Safari) ;
- `watchAudioDeviceChanges` garde l'absence d'`addEventListener` (Safari ancien) ;
- autoplay : `NotAllowedError` → `waiting_for_user` (non fatal), aucune boucle
  `play()`, lecture au geste ;
- reconnexion native : pas de double Room, pas de double piste,
  `reconnectCount++` sur `Reconnecting`, déconnexion involontaire → `error` ;
- mot de passe performer : paramètre local, jamais stocké / loggué / reflété,
  vidé de l'`<input>` après succès, transmis uniquement à `requestLiveKitToken` ;
- token LiveKit : en mémoire uniquement, jamais en snapshot / DOM / logs ;
- endpoint same-origin `/api/livekit/token`, comparaison mot de passe
  timing-safe, 401 générique (pas de distinction absent/incorrect).

## Ajouts (vs v1.0.1)

- **Control Room performer** (`/control-room`) : permission micro, sélection de
  source (BlackHole/Loopback/interface), capture, vumètre (RMS/peak/dBFS +
  clipping), gain master, diffusion LiveKit, statut ON AIR, reconnexion,
  erreurs FR, arrêt/nettoyage, diagnostic `?debug=1`.
- **Listener public** : section additive « DIRECT AUDIO » sur la page publique,
  abonnement audio uniquement, autoplay, volume/mute, reconnexion native.
- **Moteur audio local** (`src/audio/`) : permission, graphe Web Audio
  (source → gain → analyser + destination), gain, vumètre, devicechange.
- **Publisher LiveKit** (`src/audio/livekitPublisher.js`) : connecte le
  `MediaStream` post-fader à LiveKit Cloud.
- **Endpoint token serverless** (`api/livekit/token.js`) : tokens temporaires
  (room `main`, identity serveur, TTL 2 h, grants par rôle).
- **Architecture multi-entry Vite** + routeur single-entry + `vercel.json`.

## Compatibilité avec v1.0.1

Compatibilité **totale** côté métadonnées : mêmes cinq champs, même protocole
Collab-Hub, même patch Max. La couche audio est **additive** et désactivable
(`VITE_LIVEKIT_ENABLED`). La Control Room est une route séparée (`noindex`).

## Sécurité

- tokens temporaires (TTL 2 h, `Cache-Control: no-store`) ;
- secrets serveur uniquement (jamais `VITE_`) ;
- mot de passe performer jamais stocké / loggué / reflété ;
- listener sans droit de publication, performer sans droit de souscription ;
- contrôle automatique des secrets dans `npm run check`.

## Checklist GO / NO-GO pour publier v1.1.0 « latest stable »

La release **ne doit pas** être publiée comme stable tant que les items runtime
ci-dessous ne sont pas validés manuellement. Voir la checklist détaillée dans
`docs/bmad/13-livekit-stabilization-and-release.md`.

**GO si** (tous) :
- performer : Ableton → BlackHole → capture → vumètre actif → diffusion →
  ON AIR → piste `program-audio` publiée → arrêt → HORS ANTENNE (capture
  conservée) → TOUT ARRÊTER ferme AudioContext/tracks ;
- listener : clic ÉCOUTER → réception `program-audio` → lecture → mute/volume
  → couper performer → « En attente du direct » → relancer → reprise ;
- Chrome desktop ET Safari desktop : parcours complet sans erreur fatale ;
- autoplay : pas de lecture auto sans geste, `NotAllowedError` géré ;
- reconnexion réseau : RECONNEXION… puis ON AIR (ou reprise listener), pas de
  double track/Room, `reconnectCount` cohérent ;
- devicechange : état erreur, pas de crash/capture zombie/publication fantôme,
  rafraîchir + sélectionner une autre source possible ;
- sécurité : aucun token en storage, aucun password en DOM/logs après succès ;
- `npm run check` vert + CI verte sur le commit de release.

**NO-GO si** : un seul échec runtime, ou une fuite/erreur de sécurité, ou un
crash non récupérable. Documenter le défaut, le corriger, ajouter un test de
régression, recommencer la checklist.

## Limites connues

- chunk `controlRoomPage` > 500 ko (avertissement Vite, à la demande) ;
  `manualChunks` non configuré ;
- pas de refresh token (TTL 2 h ; reconnexion native suffit en session courte) ;
- changement de device en capture non supporté (sélecteur désactivé en capture) ;
- double performer : comportement actuel non protégé (les deux peuvent publier
  `program-audio` ; le listener conserve la première piste, ignore la seconde) —
  pas de protection 409 ajoutée (aucun bug critique observé) ;
- wiring de `check-livekit-secrets` dans la CI GitHub reporté ;
- vulnérabilités `npm audit` (esbuild/vite, devDeps, préexistantes) non corrigées.
